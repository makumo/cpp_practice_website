# 安全检查清单

本文档提供项目开发过程中的安全检查清单，确保代码和项目安全。

## 提交前检查清单

在每次提交代码前，请务必检查以下项目：

### 代码安全
- [ ] **敏感信息检查**
  - [ ] 确认没有硬编码的密码、API 密钥、令牌
  - [ ] 确认没有硬编码的绝对路径（特别是盘符路径）
  - [ ] 确认没有包含个人信息、邮箱、电话等

- [ ] **临时文件检查**
  - [ ] 确认没有提交临时文件（*.tmp, *.temp, *.log）
  - [ ] 确认没有提交调试代码（console.log, debugger）
  - [ ] 确认没有提交测试数据或示例数据

- [ ] **依赖检查**
  - [ ] 确认 package.json 中没有未使用的依赖
  - [ ] 确认依赖版本安全（没有已知漏洞）
  - [ ] 确认 node_modules/ 目录在 .gitignore 中

### 功能安全
- [ ] **输入验证**
  - [ ] 所有用户输入都经过验证
  - [ ] 防止 SQL 注入、XSS 攻击
  - [ ] 文件上传有大小和类型限制

- [ ] **代码执行安全**
  - [ ] 用户代码在沙箱环境中执行
  - [ ] 设置执行时间限制（当前为 5 秒）
  - [ ] 设置内存使用限制
  - [ ] 临时文件在使用后及时清理

- [ ] **权限控制**
  - [ ] API 接口有适当的权限验证
  - [ ] 敏感操作需要身份验证
  - [ ] 文件系统访问受限

### Git 安全
- [ ] **分支管理**
  - [ ] 重要修改使用特性分支
  - [ ] 不直接在 master 分支开发
  - [ ] 合并前进行代码审查

- [ ] **提交信息**
  - [ ] 提交信息清晰明确
  - [ ] 使用规范的提交格式（feat:, fix:, docs:, etc.）
  - [ ] 一次提交只包含一个功能或修复

- [ ] **文件检查**
  - [ ] 使用 `git status` 查看更改
  - [ ] 使用 `git diff` 检查具体修改
  - [ ] 确认没有误删重要文件

## 日常开发安全规范

### 文件操作
1. **删除文件前**
   - 先用 `read_file` 查看文件内容
   - 确认文件用途和重要性
   - 重要文件先备份再修改

2. **修改文件前**
   - 查看文件的完整内容
   - 理解现有代码逻辑
   - 确认修改的影响范围

3. **创建文件前**
   - 确认文件确实需要创建
   - 使用合理的文件命名
   - 将文件添加到合适的目录结构中

### 代码编写
1. **路径处理**
   - 优先使用相对路径
   - 使用 `path.join()` 或 `path.resolve()` 处理路径
   - 避免硬编码绝对路径

2. **环境变量**
   - 敏感配置使用环境变量
   - 提供 .env.example 作为示例
   - .env 文件在 .gitignore 中

3. **错误处理**
   - 所有异步操作都有错误处理
   - 不暴露详细的错误信息给用户
   - 记录错误日志供调试使用

### 测试验证
1. **修改后测试**
   - 修改后立即测试相关功能
   - 测试正常和异常情况
   - 验证没有引入新的 bug

2. **提交前测试**
   - 运行完整的测试套件
   - 检查代码编译是否通过
   - 验证功能是否正常工作

## 安全配置检查

### .gitignore 检查
- [ ] 包含 node_modules/
- [ ] 包含 .env 文件
- [ ] 包含临时文件（*.tmp, *.log）
- [ ] 包含 IDE 配置（.vscode/, .idea/）
- [ ] 包含操作系统文件（.DS_Store, Thumbs.db）

### 依赖安全
- [ ] 定期运行 `npm audit` 检查漏洞
- [ ] 及时更新有安全漏洞的依赖
- [ ] 使用 `npm shrinkwrap` 锁定依赖版本

### 编译器安全
- [ ] 用户代码使用静态编译选项
- [ ] 编译选项限制不必要的系统调用
- [ ] 设置编译超时时间

## 紧急情况处理

### 误删文件恢复
1. 使用 `git checkout` 恢复已删除的文件
2. 从 Git 历史中找回旧版本
3. 如果文件从未提交，检查临时备份

### 敏感信息泄露
1. 立即从代码中移除敏感信息
2. 更新相关的 API 密钥或密码
3. 考虑使用 `git filter-branch` 清除历史记录
4. 强制推送更新远程仓库

### 代码回滚
1. 使用 `git revert` 创建回滚提交
2. 不要使用 `git reset --hard` 修改历史
3. 在回滚前先备份当前状态

- [ ] 查看代码中的硬编码路径
- [ ] 清理临时文件和日志
- [ ] 审查最近的提交记录

### 每月检查
- [ ] 全面审查安全规范执行情况
- [ ] 更新安全检查清单
- [ ] 进行安全测试
- [ ] 检查访问日志和错误日志

### 每季度检查
- [ ] 评估整体安全策略
- [ ] 更新安全工具和流程
- [ ] 进行安全培训
- [ ] 审查和更新文档

## 最佳实践

1. **最小权限原则**
   - 只授予必要的权限
   - 使用最小权限运行服务

2. **纵深防御**
   - 多层安全防护
   - 不依赖单一安全措施

3. **安全第一**
   - 安全优先于便利性
   - 宁可多检查，不可遗漏

4. **持续改进**
   - 定期评估和改进安全措施
   - 学习最新的安全实践
